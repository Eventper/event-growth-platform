# Contract Automation Logic Clarification for Replit

Please keep the existing contracts/documents functionality, but extend the logic so that contracts are **automatically generated from the system data**.

The goal is that contracts should not need to be written manually each time.

## Contract Types That Should Be Auto-Generated

The platform should support automatic generation of:

* Client Contracts
* Vendor Contracts
* Venue Contracts
* Service Agreements
* Addendums / Amendments where needed

These should be generated from the data already stored in the system.

---

## How It Should Work

When a relevant record is created or approved, the system should be able to generate a draft contract automatically.

### Examples

### Client Contract

When an event is confirmed, the system should generate a client contract using:

* client name
* event name
* event date
* event location
* services selected
* payment terms
* company entity
* contract value

### Vendor Contract

When a vendor is selected for an event, the system should generate a vendor agreement using:

* vendor name
* vendor category
* event name
* event date
* service scope
* agreed cost
* payment terms
* company details

### Venue Contract

When a venue is confirmed, the system should generate a venue agreement using:

* venue name
* event details
* booking date
* access times
* agreed venue cost
* payment schedule
* terms and conditions

---

## Editable Before Sending

Even though contracts are auto-generated, they must still be:

* editable
* reviewable
* approvable
* sendable from inside the system

This means the workflow should be:

1. System generates contract draft automatically
2. User reviews draft
3. User edits if needed
4. User approves / finalizes
5. User sends contract from the platform
6. Contract is stored in the document record

So automation should reduce manual work, but still allow control.

---

## Sending Contracts

The platform should support sending contracts directly from the system.

That means users should be able to:

* send by email
* attach message / notes
* track sent status
* track signed / unsigned status
* resend if needed

Status examples:

* Draft
* Ready to Send
* Sent
* Viewed
* Signed
* Expired
* Cancelled

---

## Template-Based Contract System

All contracts should be generated from reusable templates.

Templates should exist for:

* client contracts
* vendor contracts
* venue contracts
* payment agreements
* addendums

The template should automatically pull in the right information from the system fields.

Example:

* client name auto-fills
* event date auto-fills
* service package auto-fills
* company details auto-fill based on entity/location

This ensures consistency and saves time.

---

## Integration With Existing Workflow

This should fit into the current workflow without replacing what is already built.

### In Pre-Planning

* client proposal generated
* client contract generated
* client contract sent

### In Vendor Management

* vendor selected
* vendor contract generated
* vendor contract reviewed and sent

### In Venue Sourcing / Confirmation

* venue selected
* venue contract generated
* venue contract reviewed and sent

## Document Repository

All generated contracts should automatically save into the Document Repository and also link back to the specific event.

That means each contract should be visible in:

* the event record
* the vendor profile if vendor-related
* the venue record if venue-related
* the central document repository
